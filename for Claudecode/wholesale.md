Без заголовка секції, без іконок (не повторювати checklist-паттерн з `/about`). Просто вступний рядок + список.

```
Незалежно від обраного рівня співпраці ви отримуєте:
```

Стиль блоку: той самий card-like вигляд, що в `<aside>` на `/locations` (`rounded-2xl border border-warm/10`, світлий кремовий фон). Новий дизайн-токен не вводити.

## 4. Форма заявки

### 4.1 Поля

| Поле | name | Тип | Обов'язкове | Валідація / коментар |
|---|---|---|---|---|
| Назва компанії / ФОП | `company` | text | так | |
| Контактна особа | `contact_name` | text | так | |
| Телефон | `phone` | tel | так | `pattern="^\+380\d{9}$"` — **той самий патерн, що вже використовується у формі чекауту в `cart.astro`**, не вигадувати новий |
| Email | `email` | email | так | Formspree підхопить як reply-to |
| Сегмент | `segment` | select | ні | Кафе / Ресторан / Готель / Інше |
| Рівень співпраці | `cooperation_level` | select | ні | Спробувати / Постійний партнер / Індивідуальні умови / Ще не визначив(-ся) — назви пунктів співпадають з заголовками карток тарифів |
| Коментар | `comment` | textarea | ні | |
| Honeypot | `_gotcha` | text, прихований | — | Офіційна конвенція Formspree: `<input type="text" name="_gotcha" style="display:none" tabindex="-1" autocomplete="off">`. Не показувати людям, не позначати як обов'язкове |

Кожне поле — зі своїм `<label>`, підписаним через `for`/`id` (той самий рівень доступності, що вже витриманий у формі чекауту).

### 4.2 Стани форми — БЕЗ `<dialog>`

Це не модалка. Форма — частина сторінки, переключення станів через `data-state` атрибут на обгортці + CSS-селектори, за тим самим принципом, що `.checkout-form-state`/`.checkout-preview-state` у `cart.astro`, але без `showModal()`, без backdrop, без фокус-трапу — просто блок, що змінює вміст на сторінці.

```html
<div id="wholesale-form-wrapper" data-state="form">
  <div class="wholesale-form-state"> ... форма ... </div>
  <div class="wholesale-sent-state" aria-live="polite"> ... підтвердження ... </div>
</div>
```

```css
#wholesale-form-wrapper[data-state="form"] .wholesale-sent-state { display: none; }
#wholesale-form-wrapper[data-state="sent"] .wholesale-form-state { display: none; }
```

**Стан "sent" (текст підтвердження):**
```
Заявку отримано.
```
Коротко, нейтрально, без обіцянок по термінах відповіді.

**Після успішного submit:** `wrapper.dataset.state = 'sent'`, фокус переноситься на текст підтвердження (той самий принцип focus management, що в чекауті — фокус іде на головний результат дії).

### 4.3 AJAX-паттерн (Formspree)

Той самий підхід, що вже використовується на існуючій контактній формі (тиждень 5): `fetch` з `Accept: application/json`, progressive enhancement — форма повинна мати робочий `action` і `method="POST"`, щоб працювати і без JS (без JS браузер відправить напряму на Formspree, користувач побачить стандартну сторінку-подяку Formspree; це прийнятний fallback, той самий, що і на контактній формі).

```js
form.addEventListener('submit', async (e) => {
  e.preventDefault();
  submitBtn.disabled = true;
  try {
    const response = await fetch(form.action, {
      method: 'POST',
      body: new FormData(form),
      headers: { 'Accept': 'application/json' }
    });
    if (response.ok) {
      wrapper.dataset.state = 'sent';
      sentHeading.focus();
    } else {
      showError();
    }
  } catch {
    showError();
  } finally {
    submitBtn.disabled = false;
  }
});
```

Кнопка submit блокується на час запиту (той самий принцип захисту від подвійного кліку, що вже застосований у `ProductsCard.astro`).

### 4.4 Помилка відправки

Мінімальний inline-текст під кнопкою submit, прихований за замовчуванням, з'являється тільки при `catch`/не-ok відповіді:

```
Не вдалося надіслати заявку. Спробуйте ще раз або зв'яжіться напряму.
```

Не робити окремий повноцінний "стан" з переключенням — досить простого показу/приховування тексту.

## 5. Технічні обмеження

- Реюзати `PageHeader.astro` для Hero — новий Hero-компонент не створювати.
- Стиль карток і aside-блоку — існуючі утиліти (`rounded-2xl`, `border-warm/10`, кремовий фон), нових дизайн-токенів не вводити.
- Тарифні дані можна тримати прямо у фронтматтері `wholesale.astro` як простий масив об'єктів — окремий файл у `src/data/` не потрібен, це не перевикористовується на інших сторінках (на відміну від `locations.ts`).
- Телефонний патерн — ідентичний чекауту, не вигадувати новий regex.
- Без нових зображень, без іконок у міні-блоці "чому з нами".
- Без `<dialog>` — весь блок форми живе на сторінці.
- `<main>` обов'язково обгортає контент сторінки (той самий стандарт, що вже застосований на `[category].astro`).
- Тексти — без тире (наскрізне правило проєкту), офіційний діловий тон, не "теплий" тон головної сторінки.
- `base` враховувати через `import.meta.env.BASE_URL`, якщо знадобляться внутрішні посилання.

## 6. Чого НЕ робити

- Не додавати іконки lucide в міні-блок "чому з нами".
- Не показувати ціну в гривнях всередині карток тарифів.
- Не дублювати дисклеймер на кожній картці — один під сіткою.
- Не робити редірект на окрему сторінку "дякуємо" — підтвердження показується inline на тій самій сторінці.
- Не використовувати `<dialog>`/`showModal()` для форми.
- Не тестувати Formspree-форму багаторазово після реального деплою.

## 7. Тестові сценарії для перевірки після реалізації (доповнення до CHECK-LIST.MD)

- Форма валідна тільки з заповненими обов'язковими полями (company, contact_name, phone, email).
- Невалідний телефон (не по патерну `+380XXXXXXXXX`) блокує submit нативною валідацією.
- Успішний submit переключає `data-state` на `sent`, фокус іде на текст підтвердження.
- Кнопка submit задизейблена під час запиту, повертається в активний стан після відповіді.
- Мережева помилка (вимкнути мережу в DevTools і спробувати сабміт) показує inline error-текст, не ламає сторінку.
- Honeypot-поле не видно візуально і не в tab-порядку (`tabindex="-1"`).
- DevTools Network → 0×404 на `/wholesale`, включно з самим сабмітом форми.
