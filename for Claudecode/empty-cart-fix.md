# Спека: fix edge case пустой корзини в чекауті

**Файл:** `cart.astro`
**Не входить в цю задачу:** `cart.astro:500` (mock telegram username → бот-заявка), `cart.astro:285` (невірний коментар). Це окремі фікси дня 6.

## Проблема

Сценарій: у вкладці A відкрито діалог чекауту (кошик був непорожній на момент відкриття). У вкладці B товар прибирають з кошика. У вкладці A натискають кнопку відправлення форми (генерація повідомлення).

Поточна логіка на цей момент: `alert()` + `dialog.close()`. Два недоліки:
1. Блокуючий нативний `alert` замість inline-повідомлення в самому діалозі.
2. Після `close()` фокус за замовчуванням намагається повернутись на кнопку-ініціатора («Оформити замовлення»), якої на сторінці вже немає — кошик порожній, і за прийнятим правилом ця кнопка в такому стані не рендериться. Фокус лишається в невизначеному місці.

## Нова логіка

**1. Перевірка — першим кроком в обробнику submit**, до валідації решти полів форми і до виклику `generateOrderMessage()` / `buildWhatsAppLink()` / `buildTelegramLink()`:

```js
if (getCart().length === 0) {
  showEmptyCartMessage();
  return;
}
```

**2. `showEmptyCartMessage()`:**
- Підміняє вміст вже відкритого діалогу на текст: «Кошик порожній. Додайте товари, щоб оформити замовлення.»
- Контейнер повідомлення — з `aria-live="polite"` (той самий патерн, що вже використаний у підтвердженні на `/wholesale`).
- Запускає `setTimeout(() => dialog.close(), 3500)`. Id таймера зберігати у змінній, доступній обробнику; при повторному виклику функції — спершу `clearTimeout` попереднього.

**3. Керування фокусом після закриття:**
- Не покладатись на дефолтну поведінку `<dialog>` (повернення фокусу на елемент-ініціатор) — цього елемента в пустому стані вже немає.
- Повісити слухач на нативну подію `close` самого діалогу:
  ```js
  dialog.addEventListener('close', () => {
    browseCatalogButton?.focus();
  });
  ```
  Це покриває і автозакриття по таймеру, і будь-яке ручне закриття (Escape, якщо воно дозволене на цьому діалозі).
- `browseCatalogButton` — CTA типу «Переглянути меню» (або як вона фактично називається/позначена), що вже є частиною порожнього стану кошика на `/cart`. **Знайти реальний елемент у розмітці, не вигадувати селектор.** Якщо в нього ще немає доступного способу послатись з JS (id/data-атрибут) — додати.

**4. Навігація не потрібна.** Діалог чекауту живе на самій `/cart`, тож закриття діалогу достатньо, щоб під ним відкрилась вже актуальна порожня корзина — вона реактивно оновлюється через `cart-updated`/`storage` (патерн тижня 7). Додатковий ре-рендер не потрібен.

## Явно поза межами цієї задачі

Сценарій «превʼю вже згенеровано, а потім кошик спорожнили в іншій вкладці» — **не баг**, поведінка прийнята свідомо: нічого не відправляється автоматично, користувач сам вирішує — закрити превʼю чи все ж надіслати повідомлення. Не чіпати.

## Перевірка після реалізації

- Вкладка A: товар у кошику, діалог чекауту відкритий. Вкладка B: прибрати товар. Вкладка A: натиснути відправлення форми → зʼявляється inline-повідомлення в діалозі, не `alert`.
- Приблизно через 3.5 с діалог закривається сам.
- Фокус після закриття — на кнопці «Переглянути меню» (чи її реальному аналозі), не втрачений.
- Ручне закриття діалогу (якщо доступне) — теж переводить фокус туди ж, не лише автозакриття.
- Повторний виклик `dialog.close()` на вже закритому діалозі — не кидає помилку в консолі.
